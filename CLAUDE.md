## What This Is

FMU Builder — a Python CLI tool that packages user-provided C source code into FMI 2.0 Co-Simulation FMUs (.fmu files). Users write a YAML config describing their C function's interface, and the tool auto-generates all FMI wrapper code, compiles with MSVC, and packages the result.

## Commands

```bash
# Install (in conda env fmu_builder)
pip install -e ".[dev]"

# Run all tests (cross-platform, no MSVC needed)
pytest tests/

# Run specific test file
pytest tests/test_config.py -v
pytest tests/test_codegen.py -v
pytest tests/test_xmlgen.py -v
pytest tests/test_e2e.py -v   # compile/FMPy tests auto-skip on non-Windows

# Build an FMU (Windows + MSVC only)
fmu-builder build examples/simple_gain/fmu_config.yaml -o output/
```

## Architecture

Three-layer C code design:

1. **fmi2_wrapper.c** (fixed, `static/`) — Implements all 27 FMI 2.0 C-API functions. Never generated. Uses a `ComponentState` struct with a flat `reals[]` array split into `[inputs | outputs | params]` sections.

2. **adapter.c** (generated via Jinja2, `templates/adapter.c.j2`) — Bridges the fixed wrapper to the user's C functions. Generated by `codegen.py` based on YAML config. Supports three call styles:
   - `individual_return`: individual args, return value maps to output
   - `individual_void`: individual args, pointer output params
   - `array`: pass entire inputs/outputs/params arrays

3. **User's C code** — Arbitrary functions; the adapter handles all bridging.

## Key Constraints

- **C89 compatible** — All C code must work with VS 2010/2012 (no `//` comments, no inline declarations, no C99 features)
- **MSVC only** — MVP uses `%VS110COMNTOOLS%` / `%VS100COMNTOOLS%` to find vcvarsall.bat
- **Real type only** — MVP only supports `fmi2Real` (double); Integer/Boolean/String stubs return `fmi2Error`
- **Jinja2 custom delimiters** — `<% %>` blocks, `<< >>` variables, `<# #>` comments (avoids C `{}` conflicts)

## Module Responsibilities

- `config.py` — Pydantic models that parse+validate YAML; cross-validates arg mappings against declared variables
- `codegen.py` — `_build_step_call()` determines call style from `FunctionArg` mappings; `generate_adapter()` renders template
- `xmlgen.py` — `_build_variables()` assigns valueReferences (inputs 0..n-1, outputs n..n+m-1, params n+m..); 1-based indices for ModelStructure
- `compiler.py` — Finds MSVC via env vars/registry, runs `cl.exe /LD /MT /O2`
- `packager.py` — ZIP with `modelDescription.xml` + `binaries/win64/<name>.dll`
- `cli.py` — Typer `build` command orchestrating the full pipeline

## Testing

- `test_config.py` — YAML validation (valid + invalid cases)
- `test_codegen.py` — adapter.c generation for all 3 call styles
- `test_xmlgen.py` — XML structure, VR ordering, output indices
- `test_e2e.py` — Full pipeline from example YAML; compile/FMPy tests `@skipif(not win32)`

## pyproject.toml

Entry point: `fmu-builder = "fmu_builder.cli:app"`. Package data includes `templates/*` and `static/**/*` (FMI headers, wrapper).
